# 会员模式与托管记忆方案

## Summary

- 保持现有 OSS core 心智不变：不登录、不提供 key 时，GoodMemory 仍可直接使用，默认只走本地模式。
- 会员能力作为产品层增量，不把订阅、登录、Stripe、远端同步直接塞回当前 `createGoodMemory()` 的 core config。
- 采用你确认的方向：全链路方案、宿主集成 slash command、本地优先同步、Markdown 用 `MEMORY.md + 主题文件`、配置里最终保存单一长期 API key。
- 关键原则：本地始终可用，会员只是“在本地之上增加云端存储、跨设备同步、pgvector/embedding 检索和托管备份”。

## Public Interfaces

- 保留现有 `createGoodMemory(config)` 不变，继续服务 OSS/local/self-hosted 场景。
- 新增一个产品层入口，建议导出为 `goodmemory/member`：
  ```ts
  createGoodMemoryMemberClient({
    workspaceRoot?: string;
    local?: {
      sqlitePath?: string;
      markdownDir?: string;
    };
    cloud?: {
      baseUrl: string;
      apiKey?: string;
    };
  })
  ```
- 新增产品层能力：
  - `auth.startLogin(): Promise<{ url: string; loginSessionId: string }>`
  - `auth.completeLogin(input): Promise<{ apiKey: string }>`
  - `auth.logout(): Promise<void>`
  - `auth.status(): Promise<{ tier: "local" | "member"; subscribed: boolean; lastSyncAt?: string }>`
  - `sync.flush(): Promise<{ pushed: number; pulled: number; pending: number }>`
  - `sync.status(): Promise<{ pending: number; lastSuccessAt?: string; degraded: boolean }>`
- slash command 不由 GoodMemory 解析；宿主产品自行把 `/login`、`/logout`、`/status` 映射到上面这些 API。
- 新增全局配置文件解析，不走项目 `.env`：
  - 默认路径：`$GOODMEMORY_CONFIG_PATH` 优先，否则 macOS/Linux 用 `~/.config/goodmemory/config.toml`，Windows 用 `%AppData%/GoodMemory/config.toml`
  - 建议格式：

    ```toml
    [cloud]
    base_url = "https://api.goodmemory.com"
    api_key = "gm_live_..."

    [defaults]
    auto_sync = true
    ```

- 本地数据默认放在工作区：
  - SQLite：`<workspaceRoot>/.goodmemory/memory.sqlite`
  - Markdown：`<workspaceRoot>/.goodmemory/memory/`

## Implementation Changes

- 本地免费模式：
  - 所有 durable memory 继续写本地 SQLite。
  - 额外引入 Markdown mirror service，把结构化记忆投影为：
    - `MEMORY.md`
    - `topics/profile.md`
    - `topics/preferences.md`
    - `topics/facts.md`
    - `topics/references.md`
    - `topics/feedback.md`
    - `topics/episodes/<yyyy-mm>.md`
  - `MEMORY.md` 只做索引，不做全量 dump。
- 会员模式写路径：
  - `remember()/feedback()/forget()/deleteAllMemory()` 先成功写本地 SQLite 和 Markdown。
  - 同时写本地 `sync_outbox`，记录幂等 `opId`、`deviceId`、`occurredAt`、`scope`、`payloadHash`。
  - 后台同步器异步 push 到云端；网络失败不影响本地使用。
- 会员模式读路径：
  - 默认热路径仍以本地 SQLite 为准，保证离线可用。
  - 云端只做补充能力：跨设备回流、远端备份、pgvector/embedding 检索、服务器侧再物化。
  - 云端不可用时自动降级为 local-only，并在 `sync.status()` 暴露 degraded 状态。
- 冲突与一致性：
  - 同步采用 mutation log，不直接拿 SQLite 文件做主写入。
  - 服务端按 `opId` 幂等应用。
  - 同一 record 多端修改时用 `updatedAt` 做 last-write-wins；`deleted/superseded/archived` 生命周期优先于 `active`。
- 云端数据分层：
  - Postgres：账户、订阅状态、结构化 memory、sync cursor、审计事件。
  - pgvector：`facts`、`references`、`episodes` 的 embedding/vector 检索。
  - Remote artifacts：版本化保存 `.md` mirror 和 `.sqlite` snapshot，作为备份/导出工件，不作为主写入源。
- 登录与订阅流程：
  - `/login` -> 宿主调用 `auth.startLogin()` -> 打开网站。
  - 网站完成注册/登录后，若未订阅则进入 Stripe 年付 `$20/year` Checkout。
  - 支付成功后，网站生成或展示 API key。
  - 自动登录路径用一次性 login code 回传本地客户端，本地再换取 API key 并写入 config。
  - 手动路径在网站账户页复制 API key，手工粘贴进 config 也能生效。
- 网站/API 最小面：
  - `POST /v1/auth/login-sessions`
  - `POST /v1/auth/exchange`
  - `GET /v1/account`
  - `POST /v1/sync/push`
  - `GET /v1/sync/pull`
  - `POST /v1/api-keys/rotate`
  - `GET /v1/exports/sqlite`
  - `GET /v1/exports/markdown`
- Stripe 事件：
  - `checkout.session.completed`
  - `customer.subscription.created|updated|deleted`
  - `invoice.paid`
  - 只有 `active` 或 `grace_period` 允许云端 sync/read；`expired` 自动回退本地模式。

## Test Plan

- 免费模式无 key 时可正常 `remember/recall/buildContext/feedback/forget`，并同时写出本地 SQLite 与 Markdown。
- 已有本地用户首次加 key 后，能把现有本地记忆做一次性 backfill 到云端。
- `/login` 浏览器流程成功后，config 自动写入 API key；手动粘贴 key 也能生效。
- 会员模式写入时即使云端断网，本地仍成功，outbox 会在恢复后补同步。
- 多设备同时修改同一记忆时，服务端按 `updatedAt + lifecycle` 收敛且不重复应用。
- `deleteAllMemory` 会同时删除云端 Postgres、pgvector、Markdown 工件和 SQLite snapshot。
- 订阅过期后客户端停止云同步但保留本地数据，不出现“因为掉会员导致本地不可用”。
- API key 撤销/轮换后，旧 key 立即失效，新 key 可无损接管同步。
- `status` 能正确显示 `local/member`、订阅状态、最后同步时间、待同步数量。

## Assumptions And Defaults

- v1 会员是个人订阅，不做团队席位和共享工作区计费。
- core 包不重新引入 billing/auth 字段；会员能力通过新增产品层入口或子模块暴露。
- 配置文件保存长期 API key，但日志、trace、CLI 输出必须默认脱敏。
- config 文件权限默认收紧为仅当前用户可读写。
- 远端 `.md` 和 `.sqlite` 是镜像/备份工件，不是服务器主库；服务器主库始终是 Postgres + pgvector。
- 本地模式和会员模式都保留 Markdown 镜像，保证用户可读、可导出、可迁移。
