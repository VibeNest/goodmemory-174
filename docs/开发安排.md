开发计划会这样排。

1.先把目标收窄到一条 canonical 能力线：coding_agent + goodmemory/ai-sdk + Codex/host。
不要一上来扩成“所有 host 都支持行为学习”。PRD 里 coding_agent 本来就是 v1 主场景，强调 runtime continuity、session journal、validated patterns 和 corrections；Phase 27 里 Codex handoff 也是唯一 gate-blocking host path，所以最合理的是先把这条线做成产品级闭环，再考虑扩面。

2.不新增一套很重的 public API，先在 adapter 层补“事件输入”契约。
我不建议现在直接公开 goodmemory/evolution。更稳的做法是：保持 createGoodMemory / remember / recall / buildContext / feedback 这条最小产品心智不变，在 goodmemory/ai-sdk 和 goodmemory/host 上加可选的 agent event ingestion，把 tool_call、tool_result、file_edit、verify_result、task_transition、user_correction 这类事件归一化后继续走现有主链路。这样既符合 PRD 的最小 API，也符合 roadmap 里“先不改 public API、等 eval gate 再决定是否公开 advanced surface”的原则。

3.把 adapter 采集到的行为事件，只编译成“选择性 evidence + experience”，绝不回退成 transcript dump。
这一步要复用现有 src/evolution 和 src/evidence/src/storage 骨架，而不是另起炉灶。规则很明确：只保留对 procedural memory 真有价值的 excerpts、失败原因、关键 verify 结果、代表性 correction，不存整段 transcript，不存大段 tool output 原文，不让 evidence plane 变成历史数据库。这个边界在 unified roadmap 里已经讲得很清楚，照着落就行。

4.把 outcome 编译成 proposal，再进入 validated_pattern，而不是让后台直接改 durable memory。
这是整个计划里最关键的工程纪律。已有 reviewer / compiler / gates / salvage 说明仓库已经选择了“proposal-driven evolution”而不是“后台直接写库”；Phase 25 也验证了 outcome-telemetry → procedural_pattern → validated_pattern 这条链在 deterministic path 上有效。所以下一步不是“让 agent 自动写更多 memory”，而是“让 agent 的成功/失败被编译成可审核、可回滚、可解释的 procedural proposals”。feedback() 仍然保留为显式人工确认入口，自动路径只负责提出和推广。

5.把 coding_agent 的 recall 真的接上这些 procedural patterns。
现在的 coding-agent example 已经有 working memory、open loops、session journal，也会用 retrievalProfile: "coding_agent" 去 recall；但下一步应该让它能额外读到 outcome-backed 的 “do / don’t / avoid / next-step” 模式，而不是只承接文本和 runtime 状态。实现上重点改 src/recall/router.ts、scoring.ts、selection.ts 这一层，让 validated patterns、open loops、session journal、relevant evidence 一起进入 planner，但仍保持 lexical-first、token 受控、buildContext 不注入原始大块内容。

6.用新的 eval gate 决定它能不能晋升成默认能力。
这一步不能靠“感觉更聪明了”。直接复用现有 eval 思路，新增两组对照：一组和当前 public text-only adapter 比，一组和 no-memory baseline 比。核心指标就盯 first_attempt_policy_adherence、failure_avoidance_rate、procedure_generalization_rate，再加上 repeated-correction 下降、Codex handoff continuity 不回退、token budget 不失控。roadmap 已经把 observe -> assist -> promote、shadow evaluation、promotion gate 写成唯一正确姿势；Phase 25 的 Layer D 指标也已经给了你最自然的第一版度量面。

7.最后才决定要不要公开 advanced surface。
我的建议是：先别急着公开一个新的 goodmemory/evolution。等上面这条线在 live provider-backed 场景里跑出稳定增益，再决定是只公开少量 adapter config，还是再开放更深的 evolution surface。这个顺序和当前状态文档、unified roadmap 是一致的：现在这些门禁和 outcome telemetry 仍属于内部能力，是否上升到正式产品面，应该由 gate 结果来决定，而不是提前拍脑袋。
