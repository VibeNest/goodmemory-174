Capability: Memory Management for LLM-Based Agents
Background
Why do we need memory management?
LLM-based agents are stateless by default: if you don't explicitly carry forward state, each turn is effectively "fresh," which breaks any workflow longer than one prompt. Good memory management is what turns "a smart autocomplete" into "an agent that can run a multi-step job."
It matters because:
Context windows are limited and expensive
Long-horizon coherence needs durable state
Personalization and "don't ask me again" UX
Accuracy + safety
Specific in Agent and prompt engineering
Prompt engineering is all about "how to control the context". Few-shot, ReAct, CoT needs dynamic facts and data.
In-context learning (ICL) is so powerful, but it has a lost-on-the-middle problem. Context-control will significantly affect the capability of ICL, and it requires short-term & long-term memory.
In Google SDK, it is called Context Engineering.
Example: personal library solution — Claude code skill for Obsidian as a long-term memory.

LangChain Memory Pattern Evolution
Time
Main Library
Solution
Problem
~2022
langchain
Memory was mostly short-term (like ConversationBufferMemory).
Scales terribly — token counts explode, costs skyrocket, LLM loses focus, AI hallucination
2024
langgraph
Checkpointer Mechanism — State: Every time an Agent executes a node, it updates a global State (usually a TypedDict or Pydantic model containing the message history and intermediate variables). Thread ID: Every unique conversation gets a unique thread_id. Persistence: After a step finishes, the Checkpointer (like MemorySaver, SqliteSaver, or PostgresSaver) saves a snapshot of the current State to a database, keyed by the thread_id.
Even with checkpointers, appending messages to a list for a year will hit the LLM's context limit. Hence langmem is designed to handle semantic and factual long-term memory.
2025
langmem
A Background Agent (an asynchronous extraction process) that runs periodically. Extraction: It reads the recent conversational State.
Synthesis: It uses an LLM to extract key facts (e.g., "User is a CS undergrad," "User prefers Python," "User is working on an AI project").
Storage: It saves these structured facts into a separate Profile database or a Vector Store.
Injection: When the user sends a new message, the system queries this Profile and injects only the relevant, condensed facts into the main Agent's System Prompt.

DB Integration
We can use HANA, MySQL, or similar DB, as long as you expose it to LangMem/LangGraph as a LangGraph BaseStore (LangGraph stores long-term memory as JSON documents under a (namespace, key) pattern).
