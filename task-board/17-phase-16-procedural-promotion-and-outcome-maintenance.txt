Phase 16: Procedural Promotion and Outcome Maintenance
======================================================

Phase Goal
----------
Turn proposal and feedback signals into reusable validated patterns while making maintenance and recall scoring consume real usage outcomes.


Status Note
-----------
- Implementation is largely closed for the core engineering slices (`T001` through `T005`).
- Phase acceptance remains open while `T006` gathers result-level evidence for stale-misuse reduction and correction-rate movement, not just trace/debug visibility.


Dependencies
------------
- Phase 15 proposal pipeline complete enough to emit governed pattern and maintenance proposals
- Phase 14 archive/evidence foundation available
- Phase 8 maintenance runner remains the orchestration entrypoint


Primary Deliverables
--------------------
- `validated_pattern` compiler over existing `FeedbackMemory`
- low-risk outcome mutation helpers such as touch/reinforce/demote semantics
- verify-driven demotion and repeated-correction repair flows
- recall scoring that consumes `accessCount`, `lastUsedAt`, and `evidenceCount`
- dream orchestration that composes hygiene plus proposal compilation
- eval slices for procedural reuse, stale misuse, and correction-rate movement


Definition of Done
------------------
- procedural reuse improves without introducing a new core durable memory taxonomy
- stale or repeatedly corrected memory can be demoted or superseded with auditable reasons
- recall ranking uses real outcome signals rather than static priority alone
- dream remains a gate/orchestrator, not a hidden high-risk mutation engine

Progress Notes
--------------
- Outcome-aware scoring now consumes recall usage plus visible evidence support and emits attribution in recall candidate traces.
- Low-risk recall touch helpers now update `accessCount`, `lastAccessedAt`, and `lastUsedAt` with a bounded window and append telemetry for the applied touches.
- Repeated verification pressure now lowers recall prominence through auditable penalty signals, while contradiction repair prefers newer or verification-safer facts and records demotion reasons.
- Accepted procedural-pattern promotions now compile idempotently into active `validated_pattern` feedback records, superseding weaker source guidance without introducing a new durable type.
- Dream orchestration now batches governed maintenance, reviewer/gate passes, and procedural compilation under cooldown and overlap protection.
- Eval traces and reports now surface maintenance-oriented debug summaries for procedural reuse, verification pressure, demotion/correction signals, and whether reuse came from compiled/governed patterns instead of direct validated feedback.
- Targeted eval replay now proves a scenario-level `accepted procedural promotion -> compiler -> validated pattern reuse` path when an accepted governed promotion is present before the final recall.
- Remaining acceptance gap: the phase still needs broader result metrics for stale-misuse reduction and correction-rate movement before it should be marked fully accepted.


Key Constraints
---------------
- Procedural memory continues to reuse `FeedbackMemory.kind = "validated_pattern"` until a richer artifact is actually needed
- high-risk durable mutations still flow through proposal/promotion gates
- outcome metrics must stay explainable in traces and debug artifacts
- no graph infra or mandatory provider dependency may be introduced here


Exit Checklist
--------------
- Validated-pattern compiler exists
- Outcome mutation helpers exist
- Verify-driven demotion exists
- Recall scoring consumes outcome signals
- Dream orchestration remains governed
- Eval slices distinguish compiled/governed procedural reuse from direct validated feedback
- Targeted scenario replay proves governed promotion can compile into reusable procedural memory
- Broader stale-misuse and correction-rate movement remains to be demonstrated before final acceptance
