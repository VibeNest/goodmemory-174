Phase 29: Bun-only Release Hardening for 0.1.0-rc.1
====================================================

Phase Goal
----------
Turn GoodMemory from a repo-proven system into a Bun-only, installable, verifiable, releasable `0.1.0-rc.1` without widening the memory surface.

This phase exists to harden and verify:

- the Bun-only prerelease package contract
- tarball-first installability for `goodmemory`, `goodmemory/ai-sdk`, and `goodmemory/host`
- installed CLI usage instead of repo-local-only CLI invocation
- release workflow and RC dry-run evidence on top of the accepted Phase 27 and Phase 28 closure

This phase does not add new memory capability, does not start `src/core` refactors, and does not widen gate-blocking host coverage beyond the accepted Codex-only Phase 27 path.


Current Status
--------------

[DONE] `package.json` is now a real prerelease package with release metadata, a strict tarball allowlist, and a real `LICENSE`.

[DONE] `gate:phase-29` is accepted and the archive/current-status/task-board evidence chain is synchronized.

Primary Deliverables
--------------------

- installed-package-first docs for `goodmemory`, `goodmemory/ai-sdk`, `goodmemory/host`, and the CLI
- tarball-installed consumer smoke proving public imports plus local-first default runtime
- installed CLI smoke proving read-only commands work from the packaged binary path
- dedicated `gate:phase-29` closure command and `release.yml` workflow
- archived RC dry-run report and archived quality-gate summary


Definition of Done
------------------
- `package.json` is no longer private and uses version `0.1.0-rc.1`
- Bun-only support is explicit in README, the reference guide, current-status, and the release checklist
- the canonical release artifact is the tarball produced by `bun pm pack`
- a fresh Bun consumer installs the tarball and uses only:
  - `goodmemory`
  - `goodmemory/ai-sdk`
  - `goodmemory/host`
- the installed CLI works through `bun run goodmemory -- ...` and a strict installed-binary smoke also passes
- tarball contents include the Bun-native source export model, docs, and CLI entrypoint without pulling in repo-only payload
- the release workflow supports:
  - `workflow_dispatch` dry runs
  - tag-triggered RC release upload
- deterministic release hardening reruns local checks while accepted Phase 27 live evidence remains archived by reference
- the canonical RC dry-run report records artifact identity, install command, runtime mode, CLI success, sqlite-vss asset/fallback outcome, and elapsed time
