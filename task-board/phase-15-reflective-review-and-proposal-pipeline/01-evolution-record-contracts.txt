Why this exists
---------------
- Proposal-driven evolution needs typed records before reviewer or maintenance code can rely on it.
- Phase 14 only established the minimum `ExperienceRecord` substrate; this slice freezes the fuller contract and adds proposal/promotion records without creating parallel audit models.
- These records must stay distinct from user-facing durable memory and recall corpora.

Write tests first
-----------------
- records reject invalid scope or missing provenance fields
- proposal types are constrained to approved categories
- export/delete behavior stays scope-aware and auditable

Implementation steps
--------------------
1. Extend the existing evolution contracts and repository access patterns in the canonical storage layer
2. Reuse existing provenance and lifecycle rules instead of creating a second audit model
3. Define the minimum linking model between proposals, evidence, and later promotion decisions

Acceptance

- Later reviewer and gate flows can build without record-shape churn
