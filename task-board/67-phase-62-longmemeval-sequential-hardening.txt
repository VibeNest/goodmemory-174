Phase 62: LongMemEval Sequential Hardening
==========================================

Status
------

[WIP] Phase 62 starts the one-benchmark-at-a-time external hardening loop with
LongMemEval.

This phase is intentionally not the final public proof report. It establishes
the LongMemEval adapter, smoke gate, and failure-intake shape first, then uses
full LongMemEval runs to drive GoodMemory quality repairs before moving to BEAM.


Driver
------

The selected external benchmark order is:

1. LongMemEval
2. BEAM
3. MemoryAgentBench
4. LoCoMo

LongMemEval goes first because it directly tests long-term chat memory across
information extraction, multi-session reasoning, knowledge updates, temporal
reasoning, and abstention.


Accepted Scope
--------------

- LongMemEval smoke adapter with synthetic shape-compatible fixtures.
- External `--benchmark-root` support for full LongMemEval data without
  vendoring upstream benchmark files.
- Baseline profiles:
  - `baseline-no-memory`
  - `baseline-full-context`
  - `goodmemory-rules-only`
  - `goodmemory-hybrid`
- Report fields for answer correctness, evidence-session recall, missed recall,
  wrong recall, abstention correctness, and question-type breakdown.
- LongMemEval miss-case taxonomy and targeted GoodMemory repairs after the
  first full run.


Out Of Scope
------------

- final public benchmark report
- BEAM, MemoryAgentBench, or LoCoMo adapters
- README-level claim that GoodMemory is already strong on LongMemEval
- vendoring full LongMemEval data
- benchmark-specific runtime hacks
- public API or public config widening

Commands
--------

```text
bun run eval:phase-62
bun run gate:phase-62
bun run eval:phase-62 -- --mode full --benchmark-root <LongMemEval data root> --max-concurrency 1
bun run eval:phase-62-full500 -- --benchmark-root <LongMemEval data root> --run-id <full-500-run-id> --resume-existing-shards
bun run eval:phase-62-full500-summary -- --run-id <full-500-run-id>
bun run eval:phase-62-full500-retry-failures -- --benchmark-root <LongMemEval data root> --source-run-id <merged-or-shard-run-id> --retry-run-id <retry-run-id> --merged-run-id <merged-run-id> --chunk-size 10 --batch-concurrency 1 --case-concurrency 1 --resume-existing-batches
bun run eval:phase-62 -- --mode full --benchmark-root <LongMemEval data root> --question-type multi-session --limit 3 --max-concurrency 1
bun run eval:phase-62-recall-diagnostic -- --mode full --benchmark-root <LongMemEval data root> --profile goodmemory-rules-only --max-concurrency 1
```

If provider failure stops a retry run normally, use the emitted merged run as
the next `--source-run-id`; it preserves successful batches and leaves only
unresolved `executionError` rows retryable. If a retry run is manually
interrupted before it can emit a merged report, rerun the same `--retry-run-id`
with `--resume-existing-batches`. The runner discovers completed
`*-batch-NNN` reports under the output directory, merges their successes, and
continues numbering from the next batch instead of requiring a manual source
list.

If a sharded full-500 run itself is interrupted after some shard reports are
written, rerun the same `--run-id` with `--resume-existing-shards`; completed
`*-shard-NN/report.json` files are reused before missing shards are scheduled.

The full command is intentionally fail-closed until it finds the external
LongMemEval data file and all env required by the selected profiles. The adapter
must not score full benchmark data from oracle labels.
