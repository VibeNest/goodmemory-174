Phase 17.5: Public Surface Decision
===================================

[DONE] P17.5-T001 Define criteria for exposing advanced config or keeping it internal
[DONE] P17.5-T002 Evaluate whether `goodmemory/evolution` or similar surfaces are warranted
[DONE] P17.5-T003 Preserve minimal OSS core when evidence is insufficient

Why this exists
---------------
- Not every internal capability should become part of the public API.
- Surface area should be earned through repeated success, not optimism.

Write tests first
-----------------
- type-level guardrails can still block unsupported config when a feature is not promoted
- release metadata/doc tests catch surface drift
- promoted surfaces align with actual supported behavior

Implementation steps
--------------------
1. Review promoted capabilities against existing public-config constraints
2. Decide which surfaces remain internal, advanced, or public
3. Resolve the official CLI shape:
   - memory-facing commands at the root
   - eval inspection nested under `goodmemory eval ...`
   - no public `goodmemory/evolution` until rollout evidence exists
4. Update type tests and release-facing docs only when the decision is real

Current implementation slice
----------------------------
1. Persist `public-surface-decision.json` and `summary.publicSurfaceDecision` from rollout evidence
2. Keep `goodmemory/evolution`, rollout config, and promotion-gate runtime controls classified as internal when evidence is insufficient
3. Add type-level guardrails so `GoodMemoryConfig` still rejects `evolution`, `strategyRollout`, `promotionGate`, and `eval` knobs
4. Add release doc checks that README and release checklist do not prematurely advertise `goodmemory/evolution`
5. Keep runtime salvage hooks off the root public surface and strip them from the root runtime wrapper so forced extra keys do not reactivate internal behavior

Acceptance
----------
- Public surface remains coherent and evidence-based
- Unsupported advanced knobs are not leaked prematurely
