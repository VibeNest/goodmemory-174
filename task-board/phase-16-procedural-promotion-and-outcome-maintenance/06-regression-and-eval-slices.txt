Why this exists
---------------
- Outcome-aware maintenance should be judged by whether it reduces bad recall and improves reuse, not by whether it mutates more records.
- These features need their own regression evidence before any default rollout.

Write tests first
-----------------
- eval can isolate validated-pattern reuse behavior
- regression outputs can show demotion/correction effects separately from baseline recall scores
- debug summaries identify which maintenance signals changed suite behavior

Implementation steps
--------------------
1. Add targeted scenario/eval slices for procedural reuse and stale suppression
2. Extend reporting to attribute gains or regressions to maintenance behavior
3. Keep the rules-only baseline available for comparison

Acceptance
----------
- Maintenance changes are measurable and reviewable
- Release decisions can rely on targeted regression evidence

Implemented Notes
-----------------
- Eval answer traces now include maintenance-oriented debug summaries for active validated patterns, verification pressure, and demotion/correction counts.
- Eval suite summaries now aggregate procedural reuse, compiled/governed procedural reuse, accepted procedural promotions, verification pressure, and demotion/correction presence across judged cases.
- Persisted eval artifacts now carry the same maintenance summary fields through case traces and top-level report output.
- Targeted eval replay now covers the scenario-level path where an accepted `procedural_pattern` promotion is compiled before the final recall and then reused as a `validated_pattern`.
- Remaining gap: the suite still lacks a broader result-level metric showing stale-misuse reduction and correction-rate movement across multiple cases, so this slice is not fully accepted yet.
