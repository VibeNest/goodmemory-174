Phase 16.4: Recall Scoring with Outcome Metrics
===============================================

Why this exists
---------------
- Better memory systems do not only rank by recency and lexical fit; they also rank by demonstrated usefulness and reliability.
- These signals should improve ranking without creating an opaque scoring model.

Write tests first
-----------------
- outcome metrics can boost stable, useful memory without displacing stronger exact matches
- demoted memories lose prominence in relevant recall slices
- scoring artifacts explain which outcome signals affected ranking

Implementation steps
--------------------
1. Extend recall scoring with bounded, interpretable weights
2. Keep lexical/policy/scope guards as hard floors
3. Emit enough attribution for eval and CLI debugging
4. Only after attribution lands, wire low-risk touch/reinforce helpers so counters can move under real recall traffic

Acceptance
----------
- Recall ranking becomes outcome-aware without becoming opaque
- Eval can attribute changes to specific scoring signals

Implemented Notes
-----------------
- Fact ranking now consumes bounded usage signals from `accessCount` and `lastAccessedAt`.
- Feedback ordering now prefers recently used guidance via `lastUsedAt`.
- Fact and reference ranking now consume visible linked-evidence counts as an explainable support signal.
- Candidate traces now expose `usageScore`, `evidenceScore`, `outcomeScore`, and `verificationPenaltyScore`.
- Verification-pressure demotion is now part of the scoring surface; remaining work is to add future correction-specific signals on top of it.
