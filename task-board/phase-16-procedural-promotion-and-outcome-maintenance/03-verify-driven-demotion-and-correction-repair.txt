Phase 16.3: Verify-Driven Demotion and Correction Repair
========================================================

[DONE] P16.3-T001 Demote stale action-driving memory after repeated verification failures
[DONE] P16.3-T002 Use repeated corrections to drive supersede/contradiction repair
[DONE] P16.3-T003 Preserve auditability for demotion and repair decisions

Why this exists
---------------
- Memory quality depends on how quickly the system backs away from stale or wrong recall.
- Repeated correction is one of the strongest signals that a current memory representation is failing.

Write tests first
-----------------
- repeated verification failures lower recall prominence without deleting unrelated history
- repeated corrections can supersede weak inferred memory in favor of better grounded content
- demotion reasons remain visible in trace and export surfaces

Implementation steps
--------------------
1. Connect verification outcomes to maintenance proposals or explicit low-risk repair rules
2. Reuse existing contradiction-repair seams from maintenance
3. Avoid direct silent deletion of valuable but stale historical evidence

Acceptance
----------
- Stale-memory misuse rate can go down through auditable maintenance behavior
- Repair logic prefers explicit, better-grounded memory over weak inferred state

Implemented Notes
-----------------
- Verification hints remain advisory on the recall path and are captured via recall/verify traces rather than silently mutating durable fact state.
- Recall scoring now applies an explainable verification-penalty signal during the current recall so stale action-driving facts lose prominence without being rewritten as failed memory.
- Durable demotion stays reserved for governed repair paths such as contradiction handling or explicit correction outcomes, where the system has stronger evidence than a hint.
- Contradiction repair now uses verification pressure plus recency tie-breaks so newer or verification-safer facts survive repair more reliably.
