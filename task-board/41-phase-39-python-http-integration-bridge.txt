Phase 39: Python HTTP Integration Bridge
========================================

Status
------
[TODO] Phase 39 is open.

This phase starts after the accepted Phase 38 governed runtime surface closure.
It does not reopen Phase 36, Phase 37, or Phase 38. It uses their accepted
public/runtime surface to make GoodMemory easier to adopt from Python/FastAPI
backends. OneLife is the first reference consumer, not the product boundary.


Intent
------
Make GoodMemory directly consumable as a generic memory middleware for
Python/FastAPI backends without coupling the library to Expo, without adding a
built-in OneLife preset, and without re-implementing any consuming app's
product policy inside the GoodMemory core.

The target integration shape is:

- Expo/Web/client apps talk only to their own Python/FastAPI backend
- the backend talks to a thin Node/Bun GoodMemory bridge over HTTP
- the bridge exposes a stable memory contract built on top of accepted public
  APIs: `recall`, `buildContext`, `remember`, `feedback`, `forget`,
  `exportMemory`, targeted `reviseMemory`, `memory.runtime.*`, and
  `memory.jobs.*`


Accepted Scope
--------------
- A thin HTTP reference contract for Python/FastAPI consumers:
  - `POST /memory/recall-context`
  - `POST /memory/remember`
  - `POST /memory/feedback`
  - `POST /memory/forget`
  - `POST /memory/export`
  - `POST /memory/revise` for targeted governed correction by explicit
    `target.memoryId`
- A Bun/Node reference bridge example built on the accepted Phase 38 runtime
  surface, not on repo-internal imports.
- A backend-only service-boundary contract: callers authenticate through the
  consuming app, and bridge requests must be scoped so export, forget, and
  revise cannot cross user/tenant boundaries.
- A reference-consumer response shape for recall-context that gives both:
  - prompt-ready context text
  - compact structured items that can map onto a consumer's current memory
    context shape; OneLife's `MemoryContextResponse` is the first reference
- Reference write-profile guidance for a `life-coach` style agent using public
  `remember` profiles, rules, annotations, and stable extractor ids.
- Explicit alignment guidance for a consumer's existing product policy and
  session/state boundaries; OneLife's `MemoryPolicyService`, `LifeOSState`, and
  `agent-session` contracts are the first reference case.
- User-control flows for view/export, delete/forget, and targeted correction by
  explicit `memoryId`; targeted revise is the bridge correction path only after
  the product has resolved a visible memory target. Consumer-side `lock` /
  "do not remember this" remains product-owned policy or masking unless it is
  intentionally mapped to an explicit `forget` or targeted `revise` outcome.
  OneLife is the reference case.
- Explicit runtime ownership guidance: the consuming app owns session lifecycle;
  the bridge can use `memory.runtime.*` only for scoped continuity snapshots or
  optional summary-only runtime journal behavior, with transcript-like archive
  persistence off by default.
- Targeted contract/regression coverage for the reference bridge and docs.


Out of Scope
------------
- Bundling GoodMemory into any client runtime
- A built-in OneLife preset inside `goodmemory`
- Replacing OneLife's `LifeOSStateService`, `MemoryPolicyService`, or business
  tables
- Root public API expansion unless the bridge/example proves a real blocker
- Transcript persistence as the default memory source
- Query-resolved correction targets or automatic memory target selection
- Bridge-managed `lock` / "do not remember this" mutations that bypass the
  consumer's product policy or visible target resolution
- Managed cloud, dashboard, hosted sync, or cross-service exactly-once claims


Acceptance Criteria
-------------------
1. A Python/FastAPI backend can integrate through a documented HTTP bridge
   without importing repo-internal GoodMemory modules.
2. The bridge contract is explicit about scope, idempotency, procedural
   feedback provenance, assistant-output confirmation, bridge-level async/sync
   controls, and user-control flows.
3. `POST /memory/recall-context` can return both prompt-ready context text and
   compact structured items for the consumer's coach path; OneLife is the first
   reference path.
4. `POST /memory/remember` supports public write customization through
   `remember` profiles/rules/annotations and does not require a built-in
   OneLife preset. Its `mode` field, if exposed, is bridge transport control
   only and is not forwarded as a new root `memory.remember()` parameter.
5. Consumer memory policy remains product-owned: blocked/shame-solidifying or
   unconfirmed identity/preference writes are filtered before durable writes,
   while GoodMemory remains the semantic memory layer; consumer-side `lock` /
   "do not remember this" policy stays outside the bridge unless mapped onto an
   explicit visible target action.
6. SQLite remains an acceptable local/single-writer path, while Postgres stays
   the recommended multi-instance production path.
7. The reference docs show how a consumer maps `userId`, `sessionId`,
   `agentId`, product state, procedural feedback, proposal/review outcomes, and
   user-control events onto GoodMemory scope and API calls; OneLife is the
   concrete example.
8. Bridge endpoints are backend-only and require caller authentication or an
   internal service boundary; export, forget, and revise require product-owned
   authorization plus scoped `userId` and tenant/workspace validation.
9. `POST /memory/revise` wraps only the Phase 38 accepted targeted
   `reviseMemory({ target: { memoryId } })` path and rejects query-resolved or
   ambiguous correction requests.
10. The reference consumer mapping is explicit: accepted/edited visible
    corrections can flow to targeted `revise`, explicit delete can flow to
    `forget`, and `lock` / "do not remember this" remains consumer-side policy
    or masking first instead of becoming a new bridge mutation.
11. OneLife owns session lifecycle; GoodMemory runtime is used only for scoped
    continuity or summary-only runtime state, and runtime archive remains off by
    default.
12. Phase 39 closes only with targeted tests, reference-consumer smoke, and a
    new immutable quality gate.
