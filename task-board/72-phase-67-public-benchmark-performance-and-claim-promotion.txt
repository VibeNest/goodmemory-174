Phase 67: Public Benchmark Performance and Claim Promotion
=========================================================

Status
------

[ACTIVE] Phase 67 turns GoodMemory's internal benchmark evidence into honest,
reproducible, public-claim-ready scores. The discipline: no internal number
reaches a public README row until it passes the public benchmark claim gate
(P67-A). This phase opens no LoCoMo retrieval work (owned by board 70) and does
not own the Phase 63 BEAM answer-pack hardening lane beyond public-claim
boundary wording; P67-D is now the BEAM public-claim re-anchor, not a request to
run more BEAM answer-generation work.


Goal
----

Turn internal benchmark evidence into honest, reproducible, public-claim-ready
scores. Every claimed benchmark must have a claim report with dataset source,
license, command, commit, model, scorer, and baseline; executionFailures 0; no
same-model judge for public semantic-judge claims; fitted AND generalization
recall reported together for BEAM.


Initial Key Finding (P67-A, 2026-06-24)
---------------------------------------

The claim gate's first verdict over all five benchmarks on 2026-06-24: 0
publicly claimable, each blocked for a precise reason.

- LongMemEval (0.908), ImplicitMemBench (0.711), BEAM (0.695): blocked by a
  SAME-MODEL judge (gpt-5.5 answer + gpt-5.5 judge; .env EVAL and JUDGE are both
  gpt-5.5 @ ai.gurkiai.com) plus unverified provenance/commit.
- MemoryAgentBench (CR 0.959 / AR 0.67): cleanest methodology — deterministic
  upstream match-mode scoring, NO judge, so no same-model bias — but blocked by
  incomplete coverage (TTL 0/30, LRU 1/6) and a missing baseline.
- LoCoMo (0.020): a retrieval boundary, not a performance result.

P67-A INFRA CONSTRAINT: there was no independent judge model available for that
first verdict (only gpt-5.5 was served). So the same-model-judge blocker on
LongMemEval/BEAM could not be cleared by a judge swap at that point. The two
viable JUDGE-FREE public-claim routes were:
  1. MemoryAgentBench — complete TTL/LRU coverage + add a baseline (P67-C); it is
     already deterministic and judge-free.
  2. LongMemEval deterministic subset — report only the judge-free methods
     (abstention/contains/exact/expected_alternative/numeric_count), excluding the
     semantic_judge questions, as a separate honest sub-score (P67-B).
Those were the fastest honest paths; the independent-judge path was infra-blocked
like LoCoMo's neural-embedding path.

[DONE 2026-07-05] P67-A2 Official-protocol rescore CLI guard. The pinned
  package command is `eval:official-rescore`; the underlying comparability
  rescore entrypoint (`scripts/rescore-official-protocols.ts`) now rejects
  duplicate scalar benchmark/source/run/judge selectors before loading
  existing LongMemEval, LoCoMo, or BEAM answer reports for official-protocol
  judging: `--benchmark`, `--reference`, `--report`, `--root`, `--rubrics`,
  `--concurrency`, `--limit`, and `--run-id`. `--run-id` must be a single path
  segment; `--concurrency` and `--limit` must be canonical positive integers.
  It also rejects benchmark-incompatible source selectors before loading reports:
  `--reference` is LongMemEval-only, `--root` is LoCoMo-only, and `--rubrics`
  is BEAM-only.
  Each output run directory now persists `run-identity.json`, rejects malformed
  or unknown-field existing identity files, rejects non-SHA-256 source
  fingerprints, and refuses to reuse an existing `progress.jsonl` cache when
  the benchmark, run id, source input paths, source input SHA-256 fingerprints,
  judge model, slice limit, or stored-answer boundary no longer match.
  Resumable progress rows now reject malformed or unknown-field cached verdicts/scores,
  malformed BEAM rubric keys whose `<questionId>#<itemIndex>` prefix disagrees
  with row `questionId`, duplicate cached question/rubric keys, and cached rows
  outside the selected case/rubric scope instead of silently counting
  non-boolean correctness, invalid BEAM rubric scores, last-write-wins summary
  drift, or stale out-of-scope slice rows, while still tolerating a killed-run
  final partial JSON line. Unit coverage now proves that tolerance for both
  question-level and BEAM rubric progress caches, while still rejecting broken
  JSON lines before the final tail.
  Generated `rescore-summary.json` files also persist `generatedAt`,
  `outputPath`, `sourceInputs`, `sourceInputFingerprints` (bytes + SHA-256),
  `judgeModel`, `limit` (number for a sliced rescore, `null` for full scope),
  `limitUnit` (`cases` for LongMemEval/LoCoMo,
  `rubric-items` for BEAM), `sourceAnswersUnchanged`, source-vs-selected scope
  counts (`sourceCases`/`selectedCases`, or BEAM `sourceQuestions` /
  `selectedQuestions` plus `sourceRubricItems` / `selectedRubricItems`), and an
  benchmark-and-judge-model-aware claim boundary for stored-answer prompt-
  compatible evidence. In particular, LongMemEval scores judged by gpt-5.4 or
  gpt-5.5 are not labeled directly comparable to published official scores.
  Source input paths are also rejected when they resolve inside the
  official-rescore output run directory, so stored answers, benchmark reference
  roots, and BEAM rubrics cannot be mixed with derived `run-identity.json`,
  `progress.jsonl`, or `rescore-summary.json` artifacts. This is input,
  source-family, run-directory, source/output isolation, malformed or
  unknown-field identity, non-SHA-256 source fingerprints, slice-budget,
  progress-cache, and source-lineage hardening for the planned independent judge
  comparability rescore only; it does not regenerate answers, run a judge in
  this guard step, change scores, or change public-claim boundaries.
  Judge gateway environment is now preflighted before creating the output run
  directory or writing `run-identity.json`: `GOODMEMORY_JUDGE_BASE_URL`,
  `GOODMEMORY_JUDGE_API_KEY`, and `GOODMEMORY_JUDGE_MODEL` must be present,
  non-empty, and free of leading/trailing whitespace. This prevents a failed
  env-only dry start from leaving a run identity with an undefined or malformed
  judge model that would later block the real rescore as identity drift.
  Final `rescore-summary.json` writes are now refused if any judge call fails;
  `progress.jsonl` stays resumable under the same run id until the selected
  scope completes without judge failures. This prevents partial official-judge
  batches from being mistaken for complete comparability evidence.
  Follow-up hardening (2026-07-06): the rescore writer now serializes
  question-level progress rows in the same strict `{correct, questionId}` shape
  that the resume parser accepts, and serializes BEAM rubric rows through the
  canonical rubric serializer covered by unit tests. This prevents a new
  LongMemEval/LoCoMo judge batch from writing `category` / raw judge text fields
  that a later strict resume would reject as unknown-field cache drift.
  `rescore-summary.json` is also schema-validated before write: stale summaries
  without stored-answer comparability metadata, source fingerprints,
  selected/source scope counts, complete judged coverage, or in-range score
  fields now fail instead of replacing auditable evidence; summaries whose
  category aggregates do not match the headline counts and scores also fail.
  The validator recomputes
  LongMemEval/LoCoMo category totals, correct counts, per-category accuracy,
  and overall accuracy; for BEAM it recomputes category question totals,
  macro-by-category, and question-weighted micro scores. The existing
  LongMemEval, LoCoMo, and BEAM official-rescore local artifacts were
  refreshed from complete cached progress with `0 to judge`: each now has a
  matching `run-identity.json`, source input SHA-256 fingerprints,
  source-vs-selected scope counts, and an explicit stored-answer comparability
  `claimBoundary` in `rescore-summary.json`, with the same published scores.

Acceptance Criteria
-------------------

- Every claimed benchmark has a claim declaration (benchmark-claims/<name>.json).
- Every claim declaration filename matches the canonical benchmark slug it
  declares.
- Every declaration has dataset source, license, command, commit, package version,
  explicit coverage, model, scorer, and baseline.
- Every declaration lists readable, non-empty repo-local evidence artifacts and
  every structured JSON artifact carries and passes non-empty claim-critical
  assertions.
- Every public declaration lists `publicClaim.readmeRequiredFragments` and
  `publicClaim.readmeDisclosureFragments`.
- Every README public-claims table row links back to the matching
  benchmark-claims/<name>.json declaration and includes that declaration's
  required score/baseline/scope fragments, while the same README includes that
  declaration's required disclosure fragments.
- Every claimable declaration appears as a row in each README public-claims table.
- executionFailures = 0.
- No same-model judge for public semantic-judge claims.
- No additional MemoryAgentBench AR/LRU scope, LoCoMo default/category scope, or
  BEAM binary-track scope is promoted unless an updated declaration passes the
  claim gate; current public scopes are MemoryAgentBench CR/TTL, LoCoMo P4
  opt-in full-10, and BEAM official-protocol 100K.
- BEAM always reports fitted and generalization recall together.
- The claim gate (gate:public-benchmark-claim --strict) is consistent (no
  over-claiming) before any README benchmark row changes.
