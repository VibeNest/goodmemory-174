Phase 28: Canonical SQLite VSS Local Backend
============================================

Phase Goal
----------
Close the remaining gap between the accepted Phase 26 local-first runtime and the stronger original local-SQLite design by making `SQLite + sqlite-vss` a real, regression-proven accelerated local semantic backend on supported runtimes.

This phase exists to harden and verify:

- real `sqlite-vss` indexed search for the local SQLite semantic path
- explicit runtime/asset bootstrap rules for supported local environments
- a clean separation between:
  - accelerated local semantic mode
  - durable local fallback mode
  - `rules-only` mode when `GOODMEMORY_EMBEDDING_*` is absent

This phase does not add bundled local embedding generation. It only upgrades the local semantic backend when embeddings are already configured and the supported local vector runtime is actually available.

Primary Deliverables
--------------------
- canonical Phase 28 contract for when local SQLite is:
  - accelerated with real `sqlite-vss`
  - durable but non-accelerated fallback
  - explicit `rules-only`
- real SQLite vector/index implementation that uses `sqlite-vss` as the accelerated backend instead of only extension-assisted scoring over the plain `vectors` table
- supported runtime/asset bootstrap contract for loading the required local SQLite vector extensions
- capability detection and diagnostics that make “accelerated” vs “fallback” explicit
- regression coverage that proves real indexed local retrieval instead of mocked or synthetic extension semantics
- dedicated `gate:phase-28` closure command and archived quality-gate summary once the runtime and evidence are real


Definition of Done
------------------
- when `GOODMEMORY_EMBEDDING_*` is configured and the supported local vector runtime is available, the local SQLite semantic path uses a real `sqlite-vss` indexed backend
- the stable local accelerated claim is backed by at least one real runtime path, not only mocked SQL functions or injected unit-test shims
- accelerated local mode and durable fallback mode are both explicit in runtime behavior and diagnostics
- `GOODMEMORY_SQLITE_VECTOR_MODE=prefer|require` continues to behave predictably:
  - `prefer` can degrade to the accepted durable fallback path
  - `require` fails loudly when accelerated local mode cannot be established
- missing `GOODMEMORY_EMBEDDING_*` still keeps runtime `rules-only`
- explicit `sqlite` / `postgres` selection and Phase 26 auto-resolution rules remain backward-compatible
- the stable product claim can distinguish supported accelerated local SQLite from the accepted non-accelerated durable fallback without overclaiming unsupported environments
