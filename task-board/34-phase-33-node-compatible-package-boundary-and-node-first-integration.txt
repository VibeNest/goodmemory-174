Phase 33: Node-Compatible Package Boundary and Node-First Integration
=====================================================================

Phase Goal
----------
Turn GoodMemory from a Bun-only, tarball-first RC into a formal Node-compatible library boundary that mainstream Node/TypeScript teams can install and integrate in 15-20 minutes without widening the memory surface.

This phase exists to harden and verify:

- standard `dist/` plus declaration outputs for `goodmemory`, `goodmemory/ai-sdk`, and `goodmemory/host`
- package metadata and export maps that no longer point the public library surface directly at `src/*.ts`
- clear isolation of Bun-only CLI behavior and Bun-local sqlite/sqlite-vss runtime paths from the Node-compatible library contract
- Node LTS plus Bun installed-package consumer smoke and CI matrix
- one canonical Node-first integration path built from `createGoodMemory()` plus `goodmemory/ai-sdk`

Primary Deliverables
--------------------
- standard build outputs and type declarations for:
  - `goodmemory`
  - `goodmemory/ai-sdk`
  - `goodmemory/host`
- package metadata and exports that resolve to built artifacts and do not require Bun-native source execution to consume the public library surface
- explicit runtime isolation for:
  - Bun-only installed CLI behavior
  - Bun-local sqlite and sqlite-vss probing/helpers
- Node LTS plus Bun installed-package consumer smoke, type smoke, and CI matrix
- one canonical Node-first integration:
  - framework-light AI SDK server path
  - public imports only

Definition of Done
------------------
- `package.json` no longer describes the main library contract as Bun-only
- `goodmemory`, `goodmemory/ai-sdk`, and `goodmemory/host` resolve through built artifacts plus `.d.ts` declarations rather than direct `src/*.ts` exports
- Node import of the canonical public surfaces does not statically pull `bun:sqlite`, `Bun.*`, or Bun-only CLI code
- Bun-only runtime concerns are isolated behind explicit runtime-specific boundaries and are not part of the gate-blocking Node library contract
- the canonical installed-package path works in:
  - Bun
  - Node LTS
- consumer smoke covers:
  - install from a packed or registry-equivalent artifact
  - runtime import and minimal remember/recall flow
  - `goodmemory/ai-sdk`
  - `goodmemory/host`
  - typecheck against emitted declarations
- CI has a public-package matrix that runs at least:
  - Node 20 LTS
  - Node 22 LTS
  - Bun current supported version
- the canonical Node-first guide and example use only public imports and no repo checkout assumptions
- canonical docs are rewritten so Bun-only CLI and Bun-local sqlite-vss support are described as runtime-specific add-ons, not as the whole product boundary
- no new memory capability, no dashboard/admin UI, no new public `goodmemory/evolution` module, and no claim of full Bun feature parity on Node where that parity is not yet real
- at least one canonical Node-first installed-consumer artifact is archived as external adoption evidence

Scope Boundary
--------------
- This phase upgrades the package/runtime boundary; it does not add new memory behavior, dashboard product surfaces, or deeper rollout families.
- This phase does not reopen accepted behavioral or host-evidence phases except where package-boundary changes require regression coverage.
- This phase does not promise Node parity for Bun-only CLI or Bun-local sqlite-vss internals before those runtime-specific paths are honestly isolated and separately documented.
- This phase keeps the canonical public memory surface centered on `createGoodMemory()` plus `goodmemory/ai-sdk` and `goodmemory/host`.
