Phase 45: First Reference Product And Adoption Evidence
=======================================================

This phase proves that GoodMemory can be adopted by a real AI product, not only
by GoodMemory's own test harness, release smoke, or installed coding-agent
dogfood.


Phase Goal
----------

Build and verify one small, runnable reference product that uses GoodMemory as a
memory layer through public package exports or the HTTP bridge, then prove an
end-to-end memory loop with product-facing evidence.

The phase should show an external developer how to:

- install GoodMemory
- start the bridge or package-backed server
- run a reference chat/copilot/backend product
- see memory improve continuity over a no-memory baseline
- inspect session, trace, and writeback audit summaries through the local viewer
- perform correction, forget, and revise through CLI/API handoff without turning
  the viewer into a write UI

Primary Product Path
--------------------

The preferred first reference product is:

```text
Reference Chat App + Python/FastAPI backend bridge + local viewer
```

The app should be intentionally small, but it must be product-shaped rather than
API-demo-only. It should have a real user conversation loop, memory-backed
context, correction/forget/revise controls, and observable traces.


Accepted Scope
--------------

- One runnable reference product under `examples/` or a similarly package-safe
  reference location.
- The reference product uses only public package exports or the `goodmemory/http`
  bridge.
- The reference product exercises:
  - `/memory/recall-context`
  - `/memory/remember`
  - `/memory/feedback`
  - `/memory/export`
  - `/memory/forget`
  - `/memory/revise`
- A product memory scenario suite with realistic chat/backend flows.
- A/B evidence comparing:
  - no-memory baseline
  - rules-only GoodMemory
  - provider-backed GoodMemory when provider env is available
- Local viewer integration that shows redacted session, trace, and writeback
  audit summaries from the reference product.
- CLI/API handoff for forget/revise remains outside the viewer.
- The reference product backend may call authenticated HTTP bridge mutation
  endpoints for product flows; the local viewer must remain read-only and may
  only generate CLI/API handoff instructions for operator review.
- Durable writes in the reference product must be explicit user/product actions
  or scripted scenario steps; after-response observe remains audit/inspectability
  unless the scenario explicitly enables selective writeback.
- Adoption docs for an external developer to install, boot, run, inspect, and
  evaluate the reference product.


Out of Scope
------------

- hosted dashboard, account system, team workspace, managed cloud, or sync
- viewer mutation routes or browser-executed forget/revise
- raw transcript archive or full assistant-output persistence
- CORS-enabled remote viewer API
- new root public API
- new installed-host hook capability
- making Claude pre-action live parity a Phase 45 blocker
- adding a large analytics system
- building another internal-only eval framework without a runnable product


Product Scenarios
-----------------

The first accepted scenario set should cover 8 to 12 flows:

1. user identity/background continuity
2. project preference continuity
3. coding style preference continuity
4. historical task continuation
5. user correction and targeted revise
6. wrong memory forget
7. procedural memory through feedback
8. observe writeback candidate visibility
9. selective writeback followed by next-turn recall
10. no-provider rules-only fallback
11. optional provider-backed retrieval uplift
12. local viewer trace/writeback/session inspection


Adoption Metrics
----------------

The adoption report should use product-facing metrics rather than only API
success counts:

- first useful recall rate
- wrong recall rate
- missed recall rate
- correction success rate
- stale memory rate
- user-visible setup steps
- time-to-first-memory-value
- observe-to-selective conversion readiness

`observe-to-selective conversion readiness` must be reported as concrete
reviewability data:

- observed candidates reviewed
- observed candidates accepted as useful
- observed candidates rejected as unsafe or noisy
- accepted/reviewed ratio
- number of scenarios where selective writeback would be justified


Acceptance Criteria
-------------------

1. A fresh external-developer path can install dependencies, start the
   GoodMemory bridge/server, and boot the reference product from documented
   commands.
2. The reference product imports only public package surfaces or calls the
   authenticated HTTP bridge.
3. The product loop proves recall, answer-context use, remember, feedback,
   export, forget, and revise.
4. At least one scenario proves user correction updates future recall behavior.
5. At least one scenario proves wrong memory can be forgotten without retaining
   raw transcript evidence in accepted artifacts.
6. The local viewer displays redacted trace/session/writeback-audit summaries
   generated by the reference product.
7. Viewer forget/revise remains CLI/API handoff only; the viewer still exposes
   no mutation routes.
8. The A/B report compares no-memory, rules-only GoodMemory, and provider-backed
   GoodMemory when provider env is available.
9. The quality gate proves no root API widening, no viewer mutation, no raw
   transcript archive, and no cloud/dashboard claim.
10. README or integration docs explain how to connect GoodMemory to a real app,
    not only how to call isolated APIs.
11. Durable write behavior is explicit: observe candidates are inspectable
    evidence, while selective writeback or direct remember/feedback/revise/forget
    happens only through explicit product actions or scripted scenario steps.
