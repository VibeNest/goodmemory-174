Driver
------

MemoryAgentBench evaluates memory agents through incremental multi-turn
interactions. Its public README describes four evaluation competencies:
Accurate Retrieval, Test-Time Learning, Long-Range Understanding, and Conflict
Resolution.

Phase 63 already exposed the two issues most likely to block Phase 64:

- broad wrong/noise growth when source-ordered evidence is appended too freely
- knowledge_update regression after recall-lift work

Phase 64 prep therefore starts with readiness analysis, not a benchmark-specific
selector patch.

Source Intake Findings (P64-T001, 2026-06-15)
---------------------------------------------

- License: MIT (upstream repo). The format and metric mapping are reusable; we
  still do NOT vendor upstream data into the repo.
- Data shape: "inject once, query multiple times" — one long source text maps to
  many questions across multi-turn interactions; the text is chunked to simulate
  conversation. Auto-downloads from HuggingFace `ai-hyz/MemoryAgentBench`
  (recsys also ships `entity2id.json`).
- Competency -> upstream datasets -> metric:
  - AR  (Accurate Retrieval):   event_qa, ruler_qa1, ruler_qa2 -> Accuracy via substring_exact_match
  - CR  (Conflict Resolution):  fact_mh, fact_sh               -> Accuracy via substring_exact_match
  - LRU (Long-Range Underst.):  detectiveQA                    -> Accuracy via exact_match
  - TTL (Test-Time Learning):   ICL_banking/clinic/nlu/trec_*  -> Accuracy via exact_match
  - Recsys:                     recsys                         -> Recall@5
  - (LongMemEval/InfBench variants use LLM-as-judge / F1; out of scope here.)
- External root: mirror the BEAM convention — read prepared cases from an
  external root (e.g. /private/tmp/MAB) via GOODMEMORY_MAB_ROOT, never from the
  repo. The adapter consumes a normalized case shape (source-text chunks +
  ordered turns + per-question competency + gold answer + match mode), not the
  raw upstream layout.
- Metric implementation: substring_exact_match (gold is a normalized substring
  of the answer, case/space-insensitive) and exact_match (normalized equality).
  Both are deterministic, so AR/CR/LRU/TTL need no LLM judge for the smoke slice.


First Smoke Findings (P64-T003/T004, 2026-06-15)
------------------------------------------------

First deterministic run of `eval:phase-64-smoke` over the four synthetic cases
(rules-only retrieval, in-memory storage, hash embedding), executionFailures 0:

- AR  : evidenceRecall 1.0, fullyRetrieved 1/1, noise 2, stale 0
- TTL : evidenceRecall 1.0, fullyRetrieved 1/1, noise 1, stale 0,
        actionPolicyTransferReady true
- LRU : evidenceRecall 1.0, fullyRetrieved 1/1, noise 3, stale 0
- CR  : evidenceRecall 1.0, fullyRetrieved 1/1, noise 1, stale 1

Reading:

- Retrieval surfaces the gold evidence for every competency, so the seed/recall
  pipeline is sound (the necessary condition for answering is met).
- CR staleSelectedCount 1: rules-only recall pulls the superseded $5,000 chunk
  alongside the current $8,000. CORRECTED FRAMING (P64-T005 investigation): this
  is NOT a bug. GoodMemory keeps the value history retrievable on purpose (BEAM
  knowledge_update / multi_session_reasoning need it); supersession retires only
  explicit-over-inferred facts. With the current value also retrieved (recall
  1.0), conflict resolution is decided at ANSWER time, not by dropping stale
  history. The live-answer scaffold confirms CR passes on the current value
  ($8,000) despite stale co-retrieval. So stale-selected is a precision signal,
  not a recall failure.
- Noise is the full non-evidence chunk set for these tiny cases (everything is
  retrieved). It confirms the noise_budgeting risk; meaningful only once larger
  cases give real distractor pressure, so it is the second repair target.
- Answer/task accuracy is intentionally not scored here (retrieval-only smoke);
  the report marks it deferred-to-live-mode for a later live generator.

Current Prep Boundary
---------------------

Phase 64 should not start by adding another source-order lexical branch. The
first useful effect-improvement preparation is to force every future
MemoryAgentBench run to report:

- answer/task accuracy
- evidence recall
- stale or superseded evidence selected
- noisy evidence selected
- action-policy transfer for learned behavior rules

This keeps Phase 64 aligned with GoodMemory's product mechanism instead of
overfitting one benchmark surface.
