Driver
------

MemoryAgentBench evaluates memory agents through incremental multi-turn
interactions. Its public README describes four evaluation competencies:
Accurate Retrieval, Test-Time Learning, Long-Range Understanding, and Conflict
Resolution.

Phase 63 already exposed the two issues most likely to block Phase 64:

- broad wrong/noise growth when source-ordered evidence is appended too freely
- knowledge_update regression after recall-lift work

Phase 64 prep therefore starts with readiness analysis, not a benchmark-specific
selector patch.

Source Intake Findings (P64-T001, 2026-06-15)
---------------------------------------------

- License: MIT (upstream repo). The format and metric mapping are reusable; we
  still do NOT vendor upstream data into the repo.
- Data shape: "inject once, query multiple times" — one long source text maps to
  many questions across multi-turn interactions; the text is chunked to simulate
  conversation. Auto-downloads from HuggingFace `ai-hyz/MemoryAgentBench`
  (recsys also ships `entity2id.json`).
- Competency -> upstream datasets -> metric:
  - AR  (Accurate Retrieval):   event_qa, ruler_qa1, ruler_qa2 -> Accuracy via substring_exact_match
  - CR  (Conflict Resolution):  fact_mh, fact_sh               -> Accuracy via substring_exact_match
  - LRU (Long-Range Underst.):  detectiveQA                    -> Accuracy via exact_match
  - TTL (Test-Time Learning):   ICL_banking/clinic/nlu/trec_*  -> Accuracy via exact_match
  - Recsys:                     recsys                         -> Recall@5
  - (LongMemEval/InfBench variants use LLM-as-judge / F1; out of scope here.)
- External root: mirror the BEAM convention — read prepared cases from an
  external root (e.g. /private/tmp/MAB) via GOODMEMORY_MAB_ROOT, never from the
  repo. The adapter consumes a normalized case shape (source-text chunks +
  ordered turns + per-question competency + gold answer + match mode), not the
  raw upstream layout.
- Metric implementation: substring_exact_match (gold is a normalized substring
  of the answer, case/space-insensitive) and exact_match (normalized equality).
  Both are deterministic, so AR/CR/LRU/TTL need no LLM judge for the smoke slice.
