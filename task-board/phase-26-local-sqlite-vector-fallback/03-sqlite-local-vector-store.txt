=====================================

Why this exists
---------------
- Auto SQLite is not enough if semantic storage still disappears between process runs.
- The current architecture promise is about local durable capability, not only about local document/session persistence.
- Platform-specific extension loading is risky enough that it needs its own slice.

Write tests first
-----------------
- SQLite vector store passes the shared `VectorStore` contract
- vector rows survive across separate memory instances
- enabling local vectors does not regress rules-only recall when embeddings are absent
- read-only contexts do not mutate local SQLite state

Implementation steps
--------------------
1. Add a real SQLite vector store
2. Make auto SQLite choose the durable vector store rather than in-memory vectors
3. Add any extension/bootstrap guards needed for future `sqlite-vss` loading
4. Keep the implementation explainable and testable without assuming platform-specific binaries

Acceptance
----------
- Local SQLite semantic storage is durable
- SQLite semantic behavior no longer depends on process-local in-memory vectors
- The design leaves a clean upgrade path to `sqlite-vss` / accelerated local search later

Current implementation notes
----------------------------
- This slice is about durability and runtime correctness first
- Local SQLite vectors now persist durably instead of falling back to process-local in-memory vectors
- `GOODMEMORY_SQLITE_CUSTOM_LIBRARY_PATH` can inject a custom SQLite library before database construction
- `GOODMEMORY_SQLITE_VECTOR_EXTENSION_PATH` can now load one or more extensions in order, which unblocks `vector0 -> vss0` style setups
- `GOODMEMORY_SQLITE_VECTOR_EXTENSION_PATH` plus optional `GOODMEMORY_SQLITE_VECTOR_EXTENSION_ENTRYPOINT` can enable extension loading
- `GOODMEMORY_SQLITE_VECTOR_MODE=prefer` swallows extension-load failures, while `require` fails loudly
- When an extension is configured and successfully loaded, SQLite search can now use an extension-backed scoring path before falling back to durable local dot-product search
- The current stable claim is:
  - durable local vectors
  - optional extension-assisted scoring/search path
- The current non-claim is:
  - full `sqlite-vss` virtual-table / Faiss-index acceleration as the canonical default backend
- That stronger indexed acceleration can still be layered on top later without changing the stable durable fallback path
