Phase 39.5: User Control, Forget, Export, and Corrections
=========================================================

Goal
----
Align GoodMemory user-control flows with a consumer's memory policy and
correction model. OneLife is the first reference case.

Scope
-----
- Document how a consumer uses `forget` and `exportMemory` for user-visible
  memory control; OneLife is the reference flow.
- Document when a consumer should use `feedback` versus `remember`.
- Document `/memory/revise` as the explicit correction endpoint for already
  resolved `memoryId` targets.
- Document the proposal-first mapping from consumer review outcomes:
  - accept/edit -> targeted `reviseMemory()`
  - delete -> `forget()`
  - lock / "do not remember this" -> consumer-side policy or masking first
- Document when a user correction should remain a consumer-side policy action
  first and when it should become a GoodMemory `reviseMemory()` or `forget()`
  follow-up.
- Document which lineage fields cross the bridge for correction or feedback
  flows, such as consumer `proposalId`, `reviewDecision`, `reason`, `evidence`,
  and `idempotencyKey`.
- Document that query-resolved target selection is out of scope; the consuming
  app must resolve the visible memory item before calling revise.
- Document authorization boundaries for view/export/delete/revise flows.

Acceptance
----------
- The docs show how to implement:
  - view/export
  - delete/forget
  - targeted revise/correct by `memoryId`
  - proposal/review outcome mapping with carried lineage
  - do-not-remember masking
  - assistant suggestion confirmation
- The contract makes it clear that raw transcript archive is out of scope.
- The contract makes it clear that consumer-side lock/do-not-remember policy can
  block writes before GoodMemory sees them and does not become an implicit new
  bridge mutation.
