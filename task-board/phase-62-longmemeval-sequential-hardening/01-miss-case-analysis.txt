Fixed Root Causes
-----------------

1. Cleaned-release schema mismatch

The cleaned LongMemEval file contains numeric `answer` values in some cases.
The adapter now normalizes numeric answers to strings instead of rejecting the
full file during validation.

2. Full-run readiness and provider isolation

Full mode now checks for external data and required env before live execution.
The rules-only profile now installs explicit no-provider adapters so project
env does not accidentally route rules-only runs through provider-backed
embedding or assisted extraction.

3. Evidence-session tracking

The first implementation inferred `retrievedSessionIds` only from rendered
context text. That lost evidence when GoodMemory correctly returned a distilled
fact. The adapter now derives retrieved sessions from GoodMemory recall records
with `sessionId`, then falls back to rendered-content matching.

4. Run isolation

Hybrid runs use Postgres, so repeated experiments with the same question id can
pollute each other. LongMemEval context building now includes `runId` in the
workspace scope.

5. Generic personal-fact extraction

The first 3-case slice exposed missing durable personal facts:

- education background:
  `I graduated with a degree in Business Administration.`
- commute duration:
  `My daily commute takes 45 minutes each way.`
- shopping app/source:
  `I use the Cartwheel app from Target.`
- coupon event:
  `I redeemed a $5 coupon on coffee creamer last Sunday.`

These are implemented as general English deterministic extraction rules, not
LongMemEval-specific prompt hacks.

6. One-off request noise filtering

The extractor was treating one-off polite requests such as `Could you please
share it with me?` and roleplay prompts such as `Please respond as the user.`
as durable procedural feedback. Those are now filtered out.

7. Full-run concurrency

Full LongMemEval execution no longer uses unbounded `Promise.all` across cases.
The runner supports `--max-concurrency` and defaults full mode to serial case
execution.

8. Fact-query guidance-lane suppression

The repaired 3-case run showed correct answers with unrelated preference and
procedural lanes still rendered for factual questions. Recall now suppresses
preference/feedback guidance lanes for direct factual recall while preserving
them for answer-style, action-driving, continuation, reference, and coding-agent
queries.

The first implementation suppressed guidance lanes by default and broke
existing product, adoption, and behavioral-adaptation evals. The accepted
repair now suppresses only explicit factual lookup queries, keeping
`How should...` guidance queries and action-adaptation probes eligible for
preference/feedback recall.

9. Full-run failure preservation

Provider 429/cooldown can interrupt broader live runs. Full-mode answer
generation and GoodMemory memory-context failures are now captured per
case/profile as `executionError` rows, preserving report output and making
provider issues distinguishable from GoodMemory recall misses.

10. Question-type sampling

The cleaned LongMemEval file is sorted by question type. The runner now accepts
repeated `--question-type` flags plus `--offset` so broader analysis can cover
multi-session, preference, temporal, update, assistant, and abstention families
without running the full 500-case suite first.

11. Personal open-loop aggregation

The 18-case diagnostic exposed that personal to-dos like `I still need to pick
up...` and `I need to return...` were either event facts or missed entirely,
and aggregate `how many` open-loop queries could only select one slot candidate.
The extractor now records pickup/return to-dos as personal identity-scoped
`open_loop` facts, and recall can return multiple lexically signaled open loops
for aggregate to-do questions. The English continuation router was also
narrowed so physical `pick up` actions do not trigger task-continuation episode
recall.

12. Verified evidence explicitness and weak lexical recall

Verified or confirmed `remember: always` annotations now make existing
annotated candidates explicit instead of leaving them inferred. Generic recall
also allows explicit facts with weak but meaningful lexical overlap to pass the
selection gate. This fixed real misses such as `58bf7951`, where the stored
fact `The play I attended... The Glass Menagerie` overlapped the query below
the old 0.2 threshold and was suppressed.

13. Compact dated user evidence

LongMemEval has-answer user turns can contain the answer-bearing statement
inside a long mixed request. The adapter now preserves sentence-level dated
facts from verified user evidence turns. This fixes broader-slice misses such
as the bedroom-wall color, tennis-racket source, and volunteered-fundraiser
date without lowering global recall thresholds further.

14. Trusted aggregate domain and ownership selection

Aggregate fact-count expansion now requires trusted evidence or a domain signal
instead of admitting generic numbered facts. `How much` money queries, medical
provider aggregation, and current ownership-count questions can retrieve
multiple relevant explicit facts without pulling unrelated numeric noise. This
fixed the bike-expense, doctor/provider, road-trip-hours, and current-bike
ownership cases in the 60-case repair loop.

15. Assistant ordinal/grouped conversation evidence

LongMemEval assistant answer turns can contain numbered answers, nested bullet
lists, and answer evidence that later questions refer to as "what did you tell
me" or "what was in the list". The adapter now preserves ordinal assistant
facts and grouped list evidence, recall prioritizes trusted conversation
evidence for previous-chat/list queries, and the answer prompt tells the model
to preserve every distinct item in grouped evidence. This closes the focused
assistant live slice at 6/6 and the 60-case live assistant family at 10/10.

16. Preference evidence and assistant follow-up preservation

Recommendation/advice queries now have a trusted conversation-evidence fallback
for explicit user preference evidence even when the wording overlap is weak.
For LongMemEval adapter ingestion, answer-bearing advice requests also preserve
the immediate assistant follow-up recommendation list, and recommendation-style
user questions are compacted into explicit interest facts. This fixes Miami
hotel feature preferences, Netflix stand-up/storytelling preferences, evening
wind-down constraints, and slow-cooker yogurt interest without adding
case-specific routing.

17. Update-series collapse and temporal/event bridge evidence

Update-like questions now prefer trusted latest evidence for repeated facts
such as mortgage preapproval amount, shared grocery-list method, and recent
family trip. LongMemEval adapter ingestion also preserves class-location facts,
pronoun-dependent bike repair expenses, sleep-before-appointment temporal
bridges, dated charity/social events, and project-role facts. These are
generic evidence-preservation and selection repairs, not case-id routing.

18. Concrete assistant-follow-up topic answer guidance

The answer generator now tells suggestion/advice answers to include concrete
assistant follow-up recommendation topics when those topics are present in
memory context. This prevents the model from collapsing remembered
recommendation lists into a vague constraint-only answer.

19. Household issue preservation and preference evidence priority

Verified user questions can contain answer-bearing household issue facts such
as a leaking faucet or scratched countertop. The LongMemEval adapter now
preserves those as explicit user evidence, and preference/advice recall
prioritizes concise compact/user evidence ahead of verbose assistant follow-up
recommendation lists. This fixed the hybrid `afdc33df` advice miss without
case-id routing.

20. Multilingual recall selection boundary

The Phase 62 selection repairs originally left several broad recall intents as
English-only helpers inside `selection.ts` and `engine.ts`. Guidance-vs-fact
suppression, aggregate count, previous-chat evidence, recommendation/advice
queries, and personal/preference evidence signals now flow through the
`LanguageService` contract. Chinese regression coverage proves aggregate money
queries and advice queries select the right Chinese facts without pulling
nearby lexical noise.
