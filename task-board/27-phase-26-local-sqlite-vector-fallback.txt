Phase 26: Local SQLite Vector Fallback
=====================================

Phase Goal
----------
Close the architecture/runtime gap by making the default runtime resolve storage and embedding the way the product intends:

- explicit `sqlite` / `postgres` always wins
- otherwise prefer `postgres` when a usable Postgres + `pgvector` target exists
- otherwise fall back to local `sqlite`
- only enable embeddings when `GOODMEMORY_EMBEDDING_*` is configured


Current Status
--------------
[DONE] Phase 26 is closed and accepted as the local-first runtime slice.
[DONE] Canonical closure evidence is archived and locked to one accepted gate run.


Dependencies
------------
- Phase 3 storage contracts
- Phase 12 embedding/vector/router plumbing
- Phase 20 through Phase 25 closure guarantees and public-surface constraints


Primary Deliverables
--------------------
- default storage resolver with explicit-over-auto precedence
- default embedding resolver driven by `GOODMEMORY_EMBEDDING_*`
- SQLite-backed vector store with shared contract parity
- explicit runtime bootstrap rules for SQLite vector extension loading
- `sqlite` storage resolution that prefers local durable vectors over in-memory vectors
- rules-only fallback when no `embeddingAdapter` exists
- regression coverage and installation/runtime docs for local semantic mode


Definition of Done
------------------
- explicit `storage.provider` remains authoritative and bypasses auto storage switching
- auto mode prefers Postgres only when a usable `pgvector` target exists; otherwise it lands on local `sqlite`
- missing `GOODMEMORY_EMBEDDING_*` keeps recall `rules-only`
- `storage.provider === "sqlite"` no longer implies `createInMemoryVectorStore()` for semantic storage
- `sqlite + embeddingAdapter + local vector runtime` can run hybrid recall with durable local vectors
- `sqlite + no embeddingAdapter` remains explicit `rules-only`
- local vector boot failure either degrades explicitly or fails fast, depending on configured mode
- Postgres behavior, trace semantics, and stable public API remain backward-compatible

Implemented So Far
------------------
- `createGoodMemory({})` now resolves to local durable SQLite by default instead of requiring explicit storage config
- explicit `storage.provider` remains authoritative
- auto mode prefers Postgres only when a connection string is provided and the target exposes `pgvector`
- `GOODMEMORY_EMBEDDING_*` now drives automatic embedding enablement; missing vars keep runtime `rules-only`
- SQLite vectors are now durable instead of process-local in-memory fallback
- SQLite runtime now supports:
  - `GOODMEMORY_SQLITE_CUSTOM_LIBRARY_PATH`
  - `GOODMEMORY_SQLITE_VECTOR_EXTENSION_PATH`
  - `GOODMEMORY_SQLITE_VECTOR_EXTENSION_ENTRYPOINT`
  - `GOODMEMORY_SQLITE_VECTOR_MODE=off|prefer|require`
- Configured SQLite vector extensions now participate in the search path, with `prefer` fallback and `require` fail-fast semantics
- CLI storage resolution now follows the same shared resolver as the library runtime

Key Constraints
---------------
- This phase solves local default storage resolution and local vector storage fallback, not bundled local embedding generation
- Do not reintroduce top-level `embedding` config into `GoodMemoryConfig`
- Do not silently claim semantic retrieval when the local vector runtime is unavailable
- On Bun/macOS, extension bootstrap must happen before the first SQLite database is opened
- Keep rules-first explainability and typed vector hooks unchanged
