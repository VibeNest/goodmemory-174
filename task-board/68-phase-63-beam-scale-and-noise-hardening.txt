Phase 63: BEAM Scale And Noise Hardening
========================================

Status
------

[WIP] Phase 63 starts the BEAM leg of the sequential external benchmark
hardening plan after Phase 62 LongMemEval closed with clean internal evidence.

This phase is internal research hardening, not the final public benchmark
report. It should carry forward the generic LongMemEval repairs while testing
GoodMemory under BEAM's larger-scale context, noise, and retrieval-stability
pressure.


Driver
------

The selected external benchmark order remains:

1. LongMemEval
2. BEAM
3. MemoryAgentBench
4. LoCoMo

BEAM goes second because LongMemEval has now proven the long-term QA and
multi-session memory loop can beat the latest accepted full-context reference
on the current internal harness. BEAM should stress the next failure mode:
large context scale, distractor/noise filtering, token budget pressure, and
recall stability under many irrelevant memories.


Accepted Scope
--------------

- Identify and record the authoritative BEAM upstream source, license, data
  access path, and benchmark task shape.
- Add small synthetic smoke fixtures that match BEAM's shape without vendoring
  full upstream data.
- Add an external `--benchmark-root` full-run adapter, following the Phase 62
  LongMemEval pattern.
- Compare at least:
  - `baseline-no-memory`
  - `baseline-full-context`
  - `goodmemory-rules-only`
  - `goodmemory-hybrid`
- Track answer correctness, evidence/session recall where applicable,
  distractor/noise selection, token/context pressure, and execution failures.
- Produce a BEAM miss-case analysis before changing GoodMemory behavior.


Out Of Scope
------------

- final public benchmark report
- MemoryAgentBench or LoCoMo adapters
- README-level claim that GoodMemory is strong on BEAM
- vendoring full BEAM data
- benchmark-specific runtime hacks
- public API or public config widening

Source Intake
-------------

- Paper:
  Tavakoli et al., "Beyond a Million Tokens: Benchmarking and Enhancing
  Long-Term Memory in LLMs", arXiv:2510.27246. The paper introduces BEAM as
  100 long coherent conversations with 2,000 validated probing questions and
  conversations up to 10M tokens.
- Dataset:
  `https://huggingface.co/datasets/Mohammadta/BEAM`
  - license: `cc-by-sa-4.0`
  - format: Parquet
  - splits visible on Hugging Face: `100K` (20 rows), `500K` (35 rows), and
    `1M` (35 rows), 90 rows total
  - useful columns include `conversation_id`, `conversation_seed`,
    `narratives`, `user_profile`, `conversation_plan`, `user_questions`,
    `chat`, and `probing_questions`
- 10M dataset:
  `https://huggingface.co/datasets/Mohammadta/BEAM-10M`
- Initial adapter strategy:
  start with the 100K split for smoke/full-run shape validation, then extend to
  500K/1M and only then decide whether 10M is feasible for local research
  runs. Keep all full upstream files external to the repo.
- Local data preparation:
  `prepare:phase-63-beam` exports the Hugging Face datasets-server rows API
  response to an external root such as `/private/tmp/BEAM/100K.json`, fails if
  rows are partial or truncated, and keeps upstream data out of git.
  When `datasets-server.huggingface.co` is unavailable, the same script also
  supports `--source github-raw`, which reconstructs the 100K export from the
  upstream GitHub conversation folders (`chat.json`, `topic.json`,
  `plan_new.txt`, `user_messages.json`,
  `probing_questions/probing_questions.json`, `labels.txt`, `main_spec.txt`,
  and `relationships.txt`). Treat GitHub-raw reruns as their own source cohort:
  the shape is complete and loader-compatible, but tiny retrieval drift can
  occur versus an older Hugging Face rows export.
