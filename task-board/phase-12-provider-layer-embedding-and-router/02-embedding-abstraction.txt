Phase 12.2: Embedding Abstraction
=================================

[DONE] P12.2-T001 Define embedding adapter contract

Why this exists
---------------
- Without embeddings, `embedding` in the architecture stays decorative.
- The first real use should be retrieval quality, not marketing.

Write tests first
-----------------
- deterministic fake embedding adapter produces stable vectors
- enabling embeddings does not break rules-only recall
- vector writes remain scope-aware

Implementation steps
--------------------
1. Add embedding adapter interface separate from eval-only generators
2. Wire embeddings into writes for durable memory types that benefit from semantic recall
3. Keep vector store optional and degrade gracefully when disabled

Acceptance
----------
- Turning embeddings on improves retrieval plumbing without changing app integration shape
- Turning embeddings off keeps the library usable

Current implementation notes
----------------------------
- `GoodMemoryConfig.adapters.embeddingAdapter` now enables semantic vector writes without exposing a top-level `embedding` runtime config surface
- remember writes embeddings for new active facts, references, and episodes after durable persistence completes
- repository vector hooks now expose typed fact/reference/episode collections with scope-aware filtering
- maintenance runner now exposes an `embeddingRepair` job that can backfill or refresh active fact/reference/episode vectors when an embedding adapter is available
- recall behavior remains rules-first for now; hybrid retrieval is deferred to later Phase 12 work
