Why this exists
---------------
- The current recall engine is query-aware, but still mostly lexical/rule-based.
- Hybrid retrieval is the first place where `embedding` creates obvious product value.

Write tests first
-----------------
- semantic hits appear only when embeddings are enabled
- lexical exact matches do not get displaced by weak semantic neighbors
- reranking does not hide provenance

Implementation steps
--------------------
1. Add per-memory-type semantic candidate collection
2. Introduce merge weights and explainable scoring
3. Keep packet budgeting and verification hints intact

Acceptance
----------

Current implementation notes
----------------------------
- recall now computes semantic candidate scores through the typed vector hooks when `strategy: "hybrid"` is requested and embedding adapters are available
- hybrid recall keeps lexical/runtime/procedural ordering guarantees as the hard floor, using semantic scores only as explainable tie-breaking input
- when hybrid routing is requested without semantic availability, routing falls back to `rules-only` and records the fallback in `routingDecision.strategyExplanation`
- eval answer packages and raw recall artifacts now preserve the resolved routing strategy, so hybrid behavior is inspectable in traces and CLI output
