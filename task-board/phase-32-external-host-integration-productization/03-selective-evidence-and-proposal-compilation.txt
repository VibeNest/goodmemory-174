Phase 32.3: Selective Evidence and Proposal Compilation
=======================================================

[DONE] Normalize accepted external agent events into selective evidence and experiences.
[DONE] Feed those experiences into the existing proposal/reviewer/compiler chain.
[DONE] Keep the automatic path proposal-driven:
- no direct canonical durable-memory mutation from host events
- no bypass around existing gates and promotion decisions
[DONE] Preserve `feedback()` as the explicit human-confirmation path.

Guardrails
----------
- Do not store full transcripts.
- Do not persist large raw tool outputs by default.
- Store only the excerpts, outcomes, corrections, and representative artifacts needed for procedural learning and continuity.
- Keep provenance explicit so every proposal can point back to host kind, session, and event excerpt.
- Before any excerpt reaches evidence, proposal input, or export surface, pass it through redaction and the applicable memory policy path.

Acceptance
----------
- repeated external host failures can generate proposal inputs through the existing internal chain
- selective event evidence is deduped and bounded
- automatic event ingestion cannot silently redefine canonical truth
- event excerpts pass `redact`, `shouldRemember`, and conflict/policy handling before they can be persisted or re-exported
- regression tests prove transcript-dump behavior does not appear in exported state or artifacts

Evidence To Add
---------------
- integration coverage around event -> evidence -> proposal plumbing
- export-memory/assertion regressions proving selective storage
- proposal/promotion lineage tests on external host events

Implemented
-----------
- integration support now hangs off `createGoodMemory()` through `src/api/integrationSupport.ts`
- public helpers now exist on:
  - `goodmemory/ai-sdk` via `ingestAgentInputEvent(...)`
  - `goodmemory/host` via `ingestHostAgentEvent(...)`
- selective event persistence now lives in `src/api/agentEventIngestion.ts`
- `tool_result`, `verify_result`, `file_edit`, `task_transition`, and selective `tool_call` inputs can persist bounded evidence/experience records
- `user_correction` now persists bounded correction evidence and then reuses the existing explicit `feedback()` path, so repeated corrections flow into the existing reviewer/compiler pipeline instead of bypassing canonical governance
- duplicate event ids now fail closed at the ingestion layer before evidence/experience duplication occurs
