Phase 18: Host Adapters and File-Authoritative Integration
==========================================================

Phase Goal
----------
Expose compiled memory artifacts to Claude/Codex-style hosts through adapters without changing GoodMemory's canonical truth model.


Status Note
-----------
- Phase 18 is now implemented and accepted.
- A dedicated host-adapter quality gate is archived in `docs/archive/quality-gates/GoodMemory-Phase-18-Quality-Gate.md` and `reports/quality-gates/phase-18/run-20260419031141/phase-18-quality-gate.json`.


Dependencies
------------
- Phase 14 archive/evidence/artifact foundation complete
- Phase 16 validated-pattern and playbook-oriented exports complete enough to be useful
- Phase 17 public surface decisions complete for any advanced adapter APIs


Primary Deliverables
--------------------
- host adapter contract for file-authoritative or file-assisted hosts
- session-memory and handoff sync flows for coding-agent hosts
- playbook/procedure artifact export for host consumption
- Claude/Codex-style integration examples and docs
- adapter safety rules for policy, scope, and provenance preservation
- parsing and structured-delta writeback for selected artifact files
- dedicated phase-18 host-adapter quality gate and archived acceptance evidence


Definition of Done
------------------
- a read-only or file-assisted host slice exists before any writable adapter mode is recommended
- hosts can consume `MEMORY.md`, `user.md`, session memory, archive recaps, and playbooks as adapter surfaces
- optional file-authoritative edit flows write back structured deltas instead of redefining core schema
- adapter writes remain policy-aware, scope-aware, and provenance-preserving
- core runtime remains usable without any host adapter
- a dedicated phase-18 gate proves the supported host surfaces, writable subset, rollback path, and examples on the current public path


Execution Strategy
------------------
Phase 18 should be executed in two steady slices instead of mixing low-risk and high-risk work:

1. Read path first
   - host adapter contract
   - session-memory and handoff sync
   - playbook/procedure artifact export
   - Claude/Codex integration examples
2. Writable path second
   - adapter safety / verification / rollback rules
   - structured-delta writeback for a minimal supported editable file subset

This keeps Claude/Codex-style integration concrete early while preventing file-authoritative writes from outrunning the policy, provenance, and rollback story.


Development Cadence
-------------------
- complete one sub-slice at a time
- tests first for each sub-slice
- run targeted regression checks after each shipped sub-slice before widening scope
- do not enable writable adapter modes until the read-path slice is stable and safety rules are in place

---------------
- structured store remains the canonical truth source in core mode
- file-authoritative behavior exists only at the adapter layer
- adapter writes must preserve scope, policy, provenance, and export/delete semantics
- unsupported host conveniences must not leak into the minimal OSS core by default


Exit Checklist
--------------

- Session-memory handoff sync exists

---------------------

- The adapter layer remains optional and does not redefine GoodMemory's structured canonical truth model.
- Reviewer and maintenance rollout remain deferred to the dedicated later phase after this host-adapter stabilization.
