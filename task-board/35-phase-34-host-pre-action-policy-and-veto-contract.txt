Phase 34: Host Pre-Action Policy and Veto Contract
==================================================

Phase Goal
----------
Turn the accepted external host line from "memory can remind" into "memory can assess, veto, and rewrite the first step" on the explicit `goodmemory/host` surface without widening the root `GoodMemory` API.

This phase exists to add one public host-level pre-action seam for the canonical `coding_agent + goodmemory + goodmemory/host + Codex` line, while keeping Claude to parity-only and leaving the root API and public `goodmemory/evolution` surface unchanged.

Definition of Done
------------------
- `goodmemory/host` publicly exports `HostActionIntent`, `HostActionAssessmentResult`, `HostActionDecision`, and `HostAdapter.assessAction()`
- root `goodmemory` API stays unchanged and no public `goodmemory/evolution` surface is added
- pre-action assessment uses the existing compiled/exported memory truth sources instead of inventing a second policy database
- memory-backed vetoes and rewrites are deterministic and fail closed
- blocked or rewritten actions leave an auditable internal record tied to `actionId`
- Codex remains the only live gate-blocking host path
- Claude gets package/bootstrap parity only and does not become a second live blocker
- deterministic/live evidence proves at least one high-risk first action is rewritten or blocked beyond the Phase 32 soft-guard path without a completion regression
